# Методичка по backend для предзащиты

Файл сделан как шпаргалка: открой его перед предзащитой и пройди сверху вниз. Главная задача - не выучить весь код наизусть, а понимать логику и уметь спокойно объяснить, что происходит в системе.

## 1. Короткий ответ: что я сделал

Backend-часть проекта - это серверная логика сервиса бронирования и покупки билетов на андеграундные концерты.

Она отвечает за:

- хранение мероприятий, пользователей, заказов и билетов в MongoDB;
- выдачу данных для сайта через JSON API;
- создание заказа и генерацию билетов;
- регистрацию и авторизацию пользователей;
- административное управление мероприятиями и заказами;
- подтверждение оплаты;
- проверку билета на входе;
- интеграцию с Telegram-ботом.

Если нужно ответить совсем коротко:

> Я отвечаю за backend API. Он принимает HTTP-запросы от сайта и Telegram-бота, проверяет данные, работает с MongoDB, создает заказы и билеты, меняет их статусы и возвращает клиентам JSON-ответы.

## 2. Архитектура простыми словами

Проект построен по клиент-серверной архитектуре.

```text
Frontend React/Vite
        |
        | HTTP JSON API
        v
Python backend
        |
        | PyMongo
        v
MongoDB

Telegram bot
        |
        | HTTP JSON API + X-Bot-Token
        v
Python backend
```

Сайт и бот не лезут в базу напрямую. Они общаются только с backend. Это важно, потому что backend является единым центром правил: он решает, можно ли купить билет, можно ли подтвердить заказ, можно ли отметить проход.

## 3. Карта backend-файлов

### `server.py`

Точка входа backend.

Что делает:

- вызывает `init_db()`, чтобы подключиться к MongoDB, создать индексы и стартовые данные;
- запускает `ThreadingHTTPServer`;
- подключает класс `Handler`, который обрабатывает HTTP-запросы.

Как объяснить:

> `server.py` запускает сервер. Перед запуском он инициализирует базу данных, а затем поднимает HTTP-сервер на порту из настроек.

### `handler.py`

Маршрутизатор API.

Что делает:

- принимает `GET`, `POST`, `PUT`, `PATCH`, `DELETE`;
- разбирает URL, например `/api/events` или `/api/orders`;
- читает JSON из тела запроса;
- проверяет токены;
- вызывает функции из `services.py`;
- возвращает JSON-ответ;
- обрабатывает ошибки через `ApiError`.

Как объяснить:

> `handler.py` - это слой HTTP. Он не содержит бизнес-логику, а только понимает, какой маршрут вызван, проверяет доступ и передает работу в `services.py`.

### `services.py`

Главная бизнес-логика.

Что делает:

- создает заказы;
- создает билеты;
- меняет статусы заказов и билетов;
- регистрирует и авторизует пользователей;
- управляет мероприятиями;
- подтверждает оплату;
- проверяет билет на входе.

Как объяснить:

> `services.py` содержит правила предметной области. Например, билет нельзя проверить, если он не подтвержден, а заказ нельзя оплатить повторно, если он уже не в статусе ожидания оплаты.

### `database.py`

Работа с MongoDB.

Что делает:

- подключается к MongoDB через `MongoClient`;
- создает индексы;
- создает администратора по умолчанию;
- создает демо-мероприятие, если база пустая;
- генерирует числовые `id` через коллекцию `counters`.

Как объяснить:

> `database.py` отделяет работу с базой от бизнес-логики. Там настраиваются коллекции, индексы и первичная инициализация данных.

### `auth.py`

Авторизация и безопасность.

Что делает:

- хеширует пароли через `PBKDF2-HMAC-SHA256`;
- добавляет соль к паролю;
- проверяет пароль через `hmac.compare_digest`;
- создает подписанный токен;
- проверяет подпись и срок действия токена.

Как объяснить:

> Пароли не хранятся в открытом виде. В базе хранится соль и хеш. Для авторизации backend выдает подписанный токен, который потом отправляется в заголовке `Authorization: Bearer ...`.

### `schemas.py`

Преобразование данных для API.

Что делает:

- превращает документы MongoDB в JSON-формат для frontend и bot;
- скрывает внутренние поля базы;
- приводит названия к frontend-формату, например `start_at` превращает в `startAt`.

Как объяснить:

> `schemas.py` нужен, чтобы база могла хранить данные в удобном backend-формате, а клиент получал аккуратный JSON.

### `config.py`

Настройки.

Что делает:

- читает `.env`;
- задает порт backend;
- задает `MONGO_URI`, `MONGO_DB`;
- читает секреты для токенов и бота.

Как объяснить:

> Настройки вынесены в переменные окружения, чтобы не менять код при переносе проекта на другой компьютер или сервер.

### `telegram_notify.py`

Уведомления администраторам в Telegram.

Что делает:

- когда пользователь сообщает, что оплатил заказ, backend может отправить сообщение администраторам;
- использует Telegram Bot API;
- берет `BOT_TOKEN` и `ADMIN_IDS` из настроек.

Как объяснить:

> Это вспомогательный модуль для уведомления администраторов о заявке на подтверждение оплаты.

### `utils.py`

Маленькие общие функции.

Что делает:

- `now_iso()` возвращает текущее время в ISO-формате;
- `make_code(prefix)` генерирует коды заказов и билетов.

Примеры кодов:

```text
OAB12CD34 - order code
T9F8A7B6C - ticket code
```

## 5. Жизненный цикл заказа и билета

Самое важное для защиты.

```text
1. Пользователь выбирает мероприятие.
2. Frontend или bot отправляет POST /api/orders.
3. Backend проверяет мероприятие, имя, фамилию, количество билетов.
4. Backend создает order со статусом AWAITING_PAYMENT.
5. Backend создает tickets с таким же статусом AWAITING_PAYMENT.
6. Пользователь переводит оплату и нажимает "оплатил".
7. Frontend или bot отправляет POST /api/orders/{orderCode}/paid.
8. Backend меняет статус на PENDING_CONFIRMATION.
9. Администратор проверяет оплату.
10. Администратор подтверждает заказ через PATCH /api/admin/orders/{orderCode}/confirm.
11. Backend меняет заказ и билеты на CONFIRMED.
12. На входе билет проверяется через POST /api/admin/tickets/{ticketCode}/check-in.
13. Backend ставит статус CHECKED_IN и записывает checked_in_at.
```

Короткая версия:

```text
AWAITING_PAYMENT -> PENDING_CONFIRMATION -> CONFIRMED -> CHECKED_IN
```

Если отменили:

```text
AWAITING_PAYMENT/PENDING_CONFIRMATION/CONFIRMED -> CANCELLED
```

## 6. Как работает создание заказа

Файл: `services.py`, функция `create_order()`.

Логика:

1. Backend ищет мероприятие по `eventId`.
2. Проверяет, что мероприятие существует.
3. Проверяет, что `event["status"] == "ACTIVE"`.
4. Проверяет имя и фамилию.
5. Проверяет количество билетов: от 1 до 5.
6. Вызывает `_insert_order()`.
7. `_insert_order()` создает документ заказа.
8. Потом создает один или несколько билетов.
9. Возвращает `order_dto()`.

Что сказать:

> При создании заказа backend сначала валидирует входные данные и состояние мероприятия. Затем создает заказ в коллекции `orders`, рассчитывает сумму как цена мероприятия умноженная на количество билетов, генерирует уникальный код заказа и создает билеты в коллекции `tickets`.

## 7. Как генерируются коды

Файлы: `utils.py`, `services.py`.

`make_code(prefix)` генерирует случайный код через `secrets.token_hex(4).upper()`.

Пример:

```text
make_code("O") -> O3A7F91B2
make_code("T") -> T8D2E4C11
```

Потом `_unique_code()` проверяет в MongoDB, что такого кода еще нет.

Что сказать:

> Код генерируется случайно криптографически более надежным модулем `secrets`, а потом проверяется на уникальность в базе. Дополнительно в MongoDB есть уникальные индексы.

## 8. Как работает оплата

В проекте оплата не списывается автоматически через банк. Логика такая:

1. Пользователь создает заказ.
2. Получает реквизиты.
3. Делает перевод.
4. Нажимает "я оплатил" и может оставить комментарий.
5. Backend меняет статус на `PENDING_CONFIRMATION`.
6. Администратор вручную подтверждает оплату.

Файл: `services.py`.

Функции:

- `mark_paid()` - пользователь сообщил, что оплатил;
- `set_order_status(..., "CONFIRMED")` - администратор подтвердил;
- `set_order_status(..., "CANCELLED")` - администратор отменил.

Что сказать:

> В дипломной версии реализована онлайн-оплата с ручным подтверждением. Это значит, что backend хранит статус оплаты и комментарий, но финальное подтверждение делает администратор после проверки платежа.

Если спросят: “Почему не автоматическая оплата?”

> Потому что цель диплома - backend-сервис бронирования и учета билетов. Интеграция с платежным провайдером выделена как направление развития. Сейчас реализован безопасный жизненный цикл заказа с ручным подтверждением.

## 9. Как работает проверка билета

Файл: `services.py`, функция `check_in()`.

Логика:

1. Backend принимает `ticketCode`.
2. Нормализует код: убирает `#`, переводит в верхний регистр.
3. Ищет билет в MongoDB.
4. Проверяет, что статус билета `CONFIRMED`.
5. Проверяет, что `checked_in_at` еще пустой.
6. Если все хорошо - ставит:

```text
status = CHECKED_IN
checked_in_at = текущее время
```

7. Возвращает данные билета.

Что сказать:

> Билет можно отметить на входе только один раз и только если он подтвержден. Это защищает от повторного прохода по одному билету.

Если спросят: “Что будет, если билет уже использовали?”

> Backend вернет ошибку `409 Conflict`, потому что статус уже не подходит или поле `checked_in_at` заполнено.

## 10. Как работает авторизация

### Регистрация

Маршрут:

```text
POST /api/auth/register
```

Backend:

1. Проверяет email, пароль, имя.
2. Хеширует пароль.
3. Создает пользователя с ролью `USER`.
4. Возвращает токен.

### Логин

Маршрут:

```text
POST /api/auth/login
```

Backend:

1. Ищет пользователя по email.
2. Проверяет пароль.
3. Возвращает подписанный токен.

### Проверка токена

Frontend хранит токен в `localStorage` под ключом:

```text
ticketing_auth_token
```

И отправляет:

```text
Authorization: Bearer <token>
```

Что сказать:

> Для обычных пользователей используется Bearer-токен. Токен подписан через HMAC и имеет срок действия 14 дней. Если подпись неверная или срок истек, backend не принимает запрос.

## 11. Как защищены admin-маршруты

Все admin-маршруты начинаются с:

```text
/api/admin/...
```

Примеры:

```text
GET /api/admin/events
POST /api/admin/events
PATCH /api/admin/orders/{orderCode}/confirm
POST /api/admin/tickets/{ticketCode}/check-in
```

Доступ возможен двумя способами:

1. Через пользователя с ролью `ADMIN`.
2. Через Telegram-бота по заголовку:

```text
X-Bot-Token: <BACKEND_BOT_TOKEN>
```

Что сказать:

> Административные операции закрыты. Сайт использует Bearer-токен администратора, а Telegram-бот использует отдельный backend-токен в заголовке `X-Bot-Token`.

## 12. Как backend связан с frontend

Файл frontend:

```text
frontend/src/app/api.ts
```

Frontend обращается к backend через `fetch`.

Базовый адрес:

```text
VITE_API_BASE_URL
```

Если переменная не задана, frontend использует:

```text
http://текущий-хост:8000/api
```

Пример:

```ts
api.events() -> GET /events
api.createOrder(...) -> POST /orders
api.confirmOrder(...) -> PATCH /admin/orders/{code}/confirm
```

Что сказать:

> Frontend не знает, как устроена база. Он вызывает функции из `api.ts`, а они отправляют HTTP-запросы на backend и получают JSON.

## 13. Как backend связан с Telegram-ботом

Файл bot:

```text
bot/backend_client.py
```

Бот вызывает backend через HTTP.

Настройки:

```text
BACKEND_API_URL=http://localhost:8000/api
BACKEND_BOT_TOKEN=...
```

Когда бот делает admin-запрос, он добавляет заголовок:

```text
X-Bot-Token: <BACKEND_BOT_TOKEN>
```

Что сказать:

> Бот является отдельным клиентом backend API. Он не работает напрямую с MongoDB, а отправляет запросы на backend. Для административных операций бот авторизуется специальным токеном.

## 15. Ошибки и HTTP-статусы

Файл: `errors.py`.

В коде используется `ApiError(status, message)`.

Примеры:

- `400 Bad Request` - неправильные данные;
- `401 Unauthorized` - нет токена или токен неверный;
- `403 Forbidden` - пользователь не админ;
- `404 Not Found` - мероприятие, заказ или билет не найден;
- `409 Conflict` - действие противоречит текущему статусу;
- `500 Internal Server Error` - непредвиденная ошибка.

Что сказать:

> Ошибки в бизнес-логике выбрасываются как `ApiError`, а `handler.py` превращает их в JSON-ответ с нужным HTTP-статусом.

## 16. Почему MongoDB

Можно ответить так:

> MongoDB выбрана потому, что backend работает с JSON-структурами, а документы MongoDB хорошо подходят для хранения событий, заказов и билетов. Также удобно быстро развивать проект: можно добавлять поля вроде `poster_url`, `payment_comment`, `telegram_username` без сложных миграций схемы.

Дополнительно:

- есть индексы для уникальности;
- удобно отдавать данные в JSON;
- PyMongo достаточно простой для учебного проекта;
- можно использовать локально или в MongoDB Atlas.

## 17. Почему `http.server`, а не FastAPI

Если спросят, не пугайся.

Ответ:

> В дипломной реализации я использовал стандартный `ThreadingHTTPServer`, чтобы явно показать обработку HTTP-запросов, маршрутизацию, чтение JSON, CORS, авторизацию и ошибки. FastAPI можно подключить в будущем, но здесь было важно показать backend-логику без скрытия ее фреймворком.

Дополнение:

> При развитии проекта можно перейти на FastAPI, чтобы получить OpenAPI-документацию, dependency injection и встроенную валидацию.

## 18. Что такое CORS и зачем он здесь

В `handler.py` в `end_headers()` добавляются заголовки:

```text
Access-Control-Allow-Origin: *
Access-Control-Allow-Headers: Content-Type, Authorization, X-Bot-Token
Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS
```

Что сказать:

> CORS нужен, чтобы frontend, запущенный на другом порту, мог обращаться к backend. Например, frontend работает на `5173`, а backend на `8000`.

## 19. Что происходит при запуске

Команда:

```powershell
cd backend
python server.py
```

Последовательность:

1. `server.py` вызывает `init_db()`.
2. `database.py` подключается к MongoDB.
3. Создаются индексы.
4. Если нет администратора, создается:

```text
admin@404.local / admin404
```

5. Если нет мероприятий, создается демо-мероприятие.
6. Запускается HTTP-сервер.

Что сказать:

> При старте backend сам готовит базу: проверяет индексы и seed-данные. Это упрощает запуск проекта на новом компьютере.

## 21. Ответы на вероятные вопросы комиссии

### Что такое backend в твоем проекте?

Backend - это серверная часть, которая принимает запросы от сайта и Telegram-бота, работает с MongoDB, выполняет бизнес-логику заказов и билетов и возвращает JSON.

### Почему база не во frontend?

Потому что frontend находится у пользователя и не должен иметь прямой доступ к базе. Все проверки, секреты, статусы и права доступа должны быть на backend.

### Как создается билет?

Когда пользователь создает заказ, backend создает запись в `orders`, а затем для каждого выбранного билета создает запись в `tickets` с уникальным `ticket_code`.

### Как узнать, что билет настоящий?

Контролер или бот отправляет код билета на backend. Backend ищет этот код в MongoDB и проверяет статус. Если билет подтвержден и еще не использован, проход разрешается.

### Что защищает от повторного прохода?

Поле `checked_in_at` и статус `CHECKED_IN`. После первой проверки backend записывает время прохода, и повторная проверка вернет ошибку.

### Как хранятся пароли?

Пароли не хранятся открытым текстом. Используется PBKDF2-HMAC-SHA256 с солью. В базе хранится только соль и хеш.

### Как работает авторизация?

После логина backend выдает подписанный HMAC-токен. Frontend отправляет его в заголовке `Authorization: Bearer ...`. Backend проверяет подпись и срок действия.

### Как бот получает доступ к admin API?

Бот отправляет заголовок `X-Bot-Token`. Backend сравнивает его с `BACKEND_BOT_TOKEN` из `.env`.

### Почему есть разные статусы?

Статусы отражают жизненный цикл заказа: ожидание оплаты, ожидание подтверждения, подтверждено, отменено, билет использован.

### Где происходит расчет суммы?

В `services.py` при создании заказа: `total_amount = price * quantity`.

### Что будет, если пользователь введет количество 100?

Backend отклонит запрос. В `create_order()` стоит ограничение от 1 до 5 билетов.

### Что будет, если мероприятие скрыто или неактивно?

Backend не даст купить билет, потому что проверяет `event["status"] == "ACTIVE"`.

### Что будет, если MongoDB недоступна?

При запуске backend пытается выполнить `ping`. Если MongoDB недоступна, backend не сможет нормально стартовать или выполнять запросы к базе.

### Зачем нужны индексы?

Индексы ускоряют поиск и гарантируют уникальность важных полей: `email_lc`, `order_code`, `ticket_code`, `id`.

### Что можно улучшить в будущем?

Можно добавить FastAPI, OpenAPI-документацию, автоматическую оплату через платежный шлюз, более подробное логирование, автотесты и резервное копирование базы.

## 22. Что выучить за 1 час

Если времени мало, учи только это:

1. Backend принимает HTTP-запросы от frontend и bot.
2. `handler.py` - маршруты и HTTP.
3. `services.py` - бизнес-логика.
4. `database.py` - MongoDB, индексы, стартовые данные.
5. `auth.py` - пароли и токены.
6. Главный цикл заказа:

```text
AWAITING_PAYMENT -> PENDING_CONFIRMATION -> CONFIRMED -> CHECKED_IN
```

7. Билет нельзя использовать дважды.
8. Admin API защищен ролью `ADMIN` или `X-Bot-Token`.
9. Пароли хранятся как PBKDF2-хеш с солью.
10. MongoDB выбрана из-за удобной работы с JSON-документами.

## 23. Мини-речь на 1 минуту

> Я отвечал за backend-часть сервиса. Backend написан на Python и предоставляет HTTP JSON API для сайта и Telegram-бота. При запуске сервер подключается к MongoDB, создает индексы и стартовые данные. Основная логика находится в `services.py`: создание мероприятий, заказов, билетов, подтверждение оплаты и проверка билета на входе. Frontend и бот не работают с базой напрямую, а отправляют запросы на backend. Для безопасности используются хеширование паролей через PBKDF2, подписанные токены для пользователей и отдельный `X-Bot-Token` для Telegram-бота. Жизненный цикл билета проходит через статусы `AWAITING_PAYMENT`, `PENDING_CONFIRMATION`, `CONFIRMED` и `CHECKED_IN`, поэтому система может контролировать оплату и не допускать повторный проход по одному билету.

## 24. Если совсем растерялся

Говори по схеме:

```text
Запрос приходит в handler.py.
handler.py определяет маршрут.
Потом вызывает services.py.
services.py проверяет правила и работает с MongoDB.
schemas.py превращает данные в JSON.
handler.py отправляет JSON обратно клиенту.
```

Эта схема подходит почти к любому вопросу “как работает”.
